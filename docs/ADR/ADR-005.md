# ADR-005 - Retreino auditavel sem retroalimentacao automatica de novos dados

- Status: aceito
- Data: 2026-05-02
- Relacionada a: [ADR-001](ADR-001.md), [ADR-004](ADR-004.md)

## Contexto

O projeto possui monitoramento batch de drift, abertura auditavel de
solicitacao de retreino e pipeline dedicado para treinar um challenger quando o
status operacional fica critico.

Entretanto, o fluxo atual nao incorpora automaticamente os novos dados
observados em producao no conjunto de treino do retreino. Na pratica:

- o drift compara a base de referencia com a base corrente
- a solicitacao de retreino e aberta corretamente
- o challenger e treinado novamente
- mas o treino continua usando a mesma base processada do treino original

Isso explica por que, em determinados cenarios, os deltas entre champion e
challenger podem ficar iguais a zero ou muito proximos de zero: o retreino
reexecuta o pipeline sobre o mesmo dataset de treino e teste, sem
retroalimentacao dos dados mais novos detectados no monitoramento.

## Confirmacao no projeto

O comportamento foi confirmado no codigo atual.

### 1. O drift usa base de referencia e base corrente diferentes

Na documentacao do fluxo de monitoramento, a base de referencia e
`data/processed/train.parquet`, enquanto a base corrente vem do log de
inferencia `artifacts/logs/inference/predictions.jsonl`.

Isso tambem aparece na rotina de drift, que prepara os insumos a partir da base
de referencia configurada e dos dados correntes observados em producao local.

### 2. A solicitacao de retreino e aberta a partir do drift

Quando o drift fica critico, a rotina cria um `retrain_request.json` com
metadados como:

- `training_config_path`
- `reference_row_count`
- `current_row_count`
- `drift_status`
- `max_feature_psi`
- `prediction_psi`

Ou seja, o monitoramento guarda o contexto do drift de forma auditavel, mas nao
anexa um novo dataset consolidado para treino.

### 3. O retreino reaproveita a configuracao de treino existente

O executor de retreino gera um arquivo temporario em
`artifacts/evaluation/model/retraining/generated_configs/`, mas esse arquivo
altera basicamente:

- `run_name`
- `version`
- `model_path`
- tags de MLflow para marcar o challenger

O fluxo nao muda a origem dos dados de treino.

### 4. O treino continua lendo os mesmos arquivos processados

O ponto central da confirmacao esta no executor de treino:

- `load_training_data(...)` le sempre `data/processed/train.parquet`
- `load_training_data(...)` le sempre `data/processed/test.parquet`
- `compute_training_data_version(...)` calcula o hash a partir desses mesmos
  dois arquivos

Assim, mesmo quando o retreino e disparado por drift, a base efetivamente usada
no challenger continua sendo a mesma base processada do treino original, salvo
se alguem atualizar esses artefatos fora do fluxo automatico.

### 5. A comparacao champion vs challenger usa metricas derivadas desse mesmo treino

Depois do retreino, a promocao compara os metadados do champion e do
challenger, calculando `metric_deltas` e `weighted_score.delta`.

Se ambos foram treinados com o mesmo dataset processado e configuracao
equivalente, os deltas podem ficar zero ou quase zero, o que e coerente com a
implementacao atual.

## Mapeamento de onde isso acontece

Os pontos principais sao:

- `src/evaluation/model/drift/drift.py`
  - detecta drift
  - gera `retrain_request.json`
  - pode disparar `run_retraining_request(...)`

- `src/model_lifecycle/retraining.py`
  - carrega a solicitacao de retreino
  - cria uma configuracao temporaria de challenger
  - reaproveita a configuracao base de treino
  - chama `run_training(...)`

- `src/model_lifecycle/train.py`
  - `load_training_data(...)` le `data/processed/train.parquet` e
    `data/processed/test.parquet`
  - `compute_training_data_version(...)` calcula hash desses mesmos arquivos

- `src/model_lifecycle/promotion.py`
  - compara champion e challenger
  - calcula `metric_deltas` e `weighted_score.delta`

- `docs/DRIFT_MONITORING.md`
  - ja documenta que o drift compara base de referencia e base corrente
  - ajuda a explicar por que a deteccao de drift existe mesmo sem
    retroalimentacao automatica no retreino

## Decisao

Aceitamos a limitacao atual: o projeto possui retreino auditavel disparado por
drift, mas ainda sem retroalimentacao automatica dos novos dados de producao no
dataset de treino.

Essa decisao e adequada para o estado atual do projeto porque prioriza:

- rastreabilidade
- auditabilidade
- separacao clara entre monitoramento e retreino
- simplicidade operacional

## Justificativa

### Por que manter assim por enquanto

Mesmo sem retroalimentacao automatica, o fluxo atual entrega valor importante:

- o drift e detectado
- o status operacional e consolidado
- a solicitacao de retreino e persistida
- o retreino e executavel e auditavel
- a comparacao champion vs challenger fica registrada

Isso permite demonstrar governanca de ML e trilha de auditoria, mesmo antes da
automacao completa do ciclo de dados.

### Por que a retroalimentacao ficou para depois

A incorporacao segura de novos dados no treino exige controles adicionais, como:

- orquestracao de jobs e tasks
- validacao de qualidade dos novos dados
- regras de janela temporal e elegibilidade
- versionamento do dataset retroalimentado
- pipeline confiavel para promover novos dados a referencia de treino

Sem um orquestrador desses jobs e tasks, automatizar essa etapa agora aumentaria
o risco de acoplamento, inconsistencias e perda de auditabilidade.

## Consequencias

### Positivas

- o fluxo de drift e retreino continua simples e auditavel
- os artefatos de monitoramento e retreino preservam rastreabilidade
- o projeto separa corretamente deteccao de drift e incorporacao de novos dados

### Negativas

- o retreino automatico nao aprende com os novos dados observados em producao
- a comparacao champion vs challenger pode gerar deltas nulos
- a resposta operacional ao drift fica incompleta do ponto de vista de
  adaptacao do modelo

## Alternativas consideradas

### Retroalimentacao automatica imediata

Adiada. Embora desejavel, essa abordagem depende de orquestracao adicional para
ser implementada com seguranca.

### Manter apenas monitoramento sem retreino

Rejeitada. O projeto ja possui um fluxo auditavel de solicitacao e execucao de
retreino que agrega valor, mesmo sem novos dados incorporados automaticamente.

## Evolucao futura

Quando houver um orquestrador de jobs e tasks, a evolucao esperada e:

- consolidar novos dados correntes em dataset elegivel para treino
- reprocessar esses dados no pipeline oficial
- versionar a nova base de treino
- disparar retreino com dados realmente atualizados
- registrar o novo `training_data_version` como evidencia da retroalimentacao

Até la, o projeto assume explicitamente que o drift e auditavel, mas a
retroalimentacao automatica de dados permanece como implementacao futura.
