# ADR-007 - Model Registry simplificado com MLflow e metadata sidecar

- Status: aceito
- Data: 2026-05-02
- Relacionada a: [ADR-001](ADR-001.md), [ADR-012](ADR-012.md)

## Contexto

O guia do Datathon espera maturidade de governanca de modelos com versionamento,
metadata obrigatoria e capacidade de rastreabilidade.

No projeto atual, essa necessidade foi atendida com:

- MLflow para tracking de runs, metricas, parametros e artefatos
- sidecars JSON de metadata para os modelos salvos
- fluxo champion/challenger auditavel
- decisao de promocao persistida em `promotion_decision.json`

Ao mesmo tempo, o projeto nao implementa um Model Registry completo com:

- approval workflow formal
- aliases de producao gerenciados por registry
- promocao entre ambientes
- rollback operacional por registry externo

## Decisao

Adotamos um model registry simplificado, baseado em:

- MLflow tracking
- artefatos locais versionados
- metadata sidecar por modelo
- comparacao auditavel champion/challenger

Esse desenho e considerado suficiente para a fase atual, sem introduzir a
complexidade de um registry operacional completo.

## Justificativa

- atende a rastreabilidade minima esperada pela banca
- preserva simplicidade operacional em ambiente local
- suporta auditoria de treino, drift e retreino
- reduz acoplamento com infraestrutura adicional

## Consequencias

### Positivas

- versionamento e metadata ficam claros e auditaveis
- o fluxo champion/challenger continua defensavel
- a implementacao permanece leve e reproduzivel

### Negativas

- nao ha approval gate formal
- nao ha registry com aliases de producao
- nao ha deploy entre ambientes integrado ao registry

## Evolucao futura

Quando houver necessidade de maior maturidade operacional, a evolucao natural e:

- introduzir aliases de producao
- adicionar approval workflow
- integrar promocao com ambientes staging/producao
- formalizar rollback via registry
