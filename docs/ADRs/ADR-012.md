# ADR-012 - Indice vetorial em memoria com cache local, sem vector store externo

- Status: aceito
- Data: 02-05-2026
- Relacionada a: [ADR-003](ADR-003.md)

## Contexto

O RAG do projeto indexa um corpus pequeno e controlado, formado por
`README.md`, arquivos `docs/**/*.md` e poucos JSON relevantes definidos em
codigo. Nesse contexto, foi necessario decidir entre usar um vector store
dedicado ou manter um indice vetorial local com cache persistido.

## Decisao

Adotamos um desenho de RAG sem vector store externo. O projeto:

- gera embeddings localmente
- persiste o indice vetorial em cache local
- recarrega esse indice em memoria no processo do serving
- reconstrui o cache apenas quando o manifesto das fontes muda

## Justificativa

Para o tamanho atual do corpus, um vector store dedicado aumentaria a
complexidade operacional sem ganho proporcional. O desenho atual atende bem
porque:

- o volume de documentos e pequeno
- o startup pode reaproveitar cache local em disco
- a consulta vetorial em memoria e suficiente para a latencia esperada
- a operacao local fica mais simples para desenvolvimento, demo e banca
- ha menos dependencias, containers e pontos extras de falha

O cache local evita recomputar embeddings e reconstruir todo o indice a cada
reinicio. Quando as fontes nao mudam, o projeto reaproveita o artefato salvo em
`joblib`, reduzindo custo de inicializacao e mantendo comportamento previsivel.

## Consequencias

- menor complexidade arquitetural
- menor custo operacional local
- inicializacao mais rapida quando ha `cache hit`
- menos dependencias de infraestrutura para a trilha RAG
- comportamento facil de explicar e auditar
- o desenho nao e o mais indicado para corpus muito maiores
- nao ha servico vetorial compartilhado entre multiplas instancias
- atualizacoes dinamicas do indice sao mais limitadas do que em plataformas
  vetoriais dedicadas
- uma evolucao para alta concorrencia ou multi-tenant pode exigir revisao

## Alternativas consideradas

Adiado. Solucoes como Chroma, Qdrant, Milvus ou equivalentes poderiam ser
consideradas no futuro, mas hoje introduziriam mais custo de operacao do que
beneficio real para o corpus do projeto.

Rejeitada. O projeto precisava de recuperacao semantica com embeddings para
apoiar melhor o agente em perguntas documentais e reduzir dependencia de match
lexical exato.

## Relacao com a ADR-003

A [ADR-003](ADR-003.md) registra a decisao de usar agente ReAct com `rag_search`
como tool documental prioritaria.

Esta ADR complementa aquela decisao ao explicitar como a camada de RAG e
materializada:

- a ADR-003 define o papel do RAG dentro do agente
- a ADR-012 define por que o indice vetorial nao usa vector store externo

## Evolucao futura

Essa decisao deve ser reavaliada se houver mudanca relevante de contexto, como:

- crescimento expressivo do corpus documental
- necessidade de atualizacao mais frequente e dinamica do indice
- multiplas instancias compartilhando a mesma base vetorial
- aumento importante de concorrencia
- necessidade de filtros, particionamento ou operacao multi-tenant
