# ADR-004 - Estrategia de avaliacao em camadas para modelo, operacao e trilha LLM

- Status: aceito
- Data: 02-05-2026
- Relacionada a: [ADR-002](ADR-002.md), [ADR-003](ADR-003.md)

## Contexto

O projeto nao possui apenas um modelo supervisionado de churn. Ele combina:

- modelo tabular para predicao
- cenarios de negocio para simulacao
- monitoramento de drift e retreino auditavel
- agente ReAct com RAG e tools de dominio

Como essas camadas tem objetivos diferentes, uma avaliacao unica seria
insuficiente para defender tecnicamente a qualidade do sistema.

O repositorio ja possui evidencias concretas de avaliacao em multiplas frentes:

- metricas supervisionadas do modelo tabular
- cenarios controlados de inferencia
- monitoramento operacional com drift e retreino
- golden set para a trilha LLM
- RAGAS com quatro metricas
- LLM-as-judge com criterio de negocio
- benchmark A/B de prompts

Essa estrategia tambem depende das decisoes anteriores:

- a [ADR-002](ADR-002.md) define que o provider LLM primario e `Claude`
- a [ADR-003](ADR-003.md) define que a camada conversacional usa agente ReAct
  com tools e RAG local

## Decisao

Adotamos uma estrategia de avaliacao em camadas, separando explicitamente as
seguintes frentes:

1. avaliacao supervisionada do modelo tabular
2. avaliacao por cenarios de negocio
3. avaliacao operacional de drift e retreino
4. avaliacao generativa da trilha RAG/LLM

Cada camada deve ser avaliada com metricas, artefatos e fluxos proprios, sem
misturar criterios de qualidade de naturezas diferentes.

Para a trilha generativa, a decisao adicional e que a avaliacao deve ocorrer
preferencialmente sobre o endpoint real `POST /llm/chat`, passando pelo mesmo
fluxo de provider, agente, tools, guardrails e RAG usado na operacao.

## Justificativa

### Por que avaliar em camadas

O projeto tem componentes com comportamentos distintos:

- o modelo tabular responde por performance preditiva
- os cenarios verificam coerencia de negocio
- o drift mede saude operacional ao longo do tempo
- o agente LLM mede qualidade de recuperacao, grounding e utilidade da resposta

Misturar tudo em uma unica nota reduziria a clareza da defesa tecnica e da
auditoria. A separacao em camadas deixa mais facil responder:

- se o modelo esta bom
- se o comportamento faz sentido para o negocio
- se a operacao continua saudavel
- se a camada conversacional responde com qualidade

### Camada 1: modelo tabular

O modelo supervisionado e avaliado com metricas classicas de classificacao,
incluindo:

- `auc`
- `f1`
- `precision`
- `recall`
- `accuracy`

Essa camada sustenta a comparacao entre experimentos, o fluxo
champion-challenger e a promocao de modelo.

### Camada 2: cenarios de negocio

Os cenarios validam se a inferencia se comporta de forma coerente em casos
controlados do dominio bancario. Essa camada e importante porque nem toda
avaliacao de negocio pode ser reduzida a metricas agregadas de classificacao.

Ela complementa o modelo supervisionado com verificacoes mais interpretaveis
para banca, produto e operacao.

### Camada 3: drift, retreino e operacao

A camada operacional responde se o sistema continua aderente ao dominio usado no
treino. Nela entram:

- PSI por feature
- PSI das probabilidades previstas
- classificacao de drift
- gatilhos para retreino
- decisao auditavel de promocao

Essa frente e necessaria porque um modelo bom no treino pode degradar em
producao local ou em dados correntes.

### Camada 4: trilha RAG/LLM

A avaliacao generativa precisa capturar dimensoes que nao aparecem nas metricas
tabulares, como grounding, qualidade do retrieval e utilidade da resposta.

Por isso, adotamos quatro componentes complementares:

- golden set do dominio
- RAGAS
- LLM-as-judge
- benchmark A/B de prompts

## Estrategia escolhida para a trilha generativa

### Golden set

O golden set define perguntas e referencias relevantes ao dominio, incluindo:

- churn bancario
- MLOps
- API
- observabilidade
- agente e RAG

Ele funciona como base comum para comparacao reprodutivel.

### RAGAS

O RAGAS mede quatro dimensoes tecnicas:

- `faithfulness`
- `answer_relevancy`
- `context_precision`
- `context_recall`

Essas metricas ajudam a separar problemas de grounding, relevancia da resposta
e qualidade do retrieval.

### LLM-as-judge

O LLM-as-judge complementa o RAGAS com criterios mais proximos da utilidade da
resposta. No projeto, os criterios sao:

- `adequacao_negocio`
- `correcao_conteudo`
- `clareza_utilidade`

O criterio `adequacao_negocio` registra explicitamente o alinhamento ao dominio.

### Benchmark A/B de prompts

O benchmark de prompts compara variantes de instrucoes sobre o mesmo golden set
para apoiar decisoes de melhoria da camada conversacional sem alterar
imediatamente a arquitetura do agente.

## Relacao com ADR-002 e ADR-003

### Relacao com ADR-002

A escolha do `Claude` como provider primario afeta a avaliacao generativa,
porque:

- influencia a estabilidade das respostas
- influencia custo e latencia das avaliacoes
- influencia a aderencia ao contrato de JSON do agente

Se o provider primario mudar no futuro, os resultados de avaliacao podem mudar
mesmo sem alteracoes no agente ou no corpus.

### Relacao com ADR-003

A avaliacao generativa depende diretamente da arquitetura do agente ReAct,
porque o endpoint `/llm/chat` passa por:

- selecao de tools
- uso de `rag_search`
- uso de `predict_churn`
- uso de `drift_status`
- uso de `scenario_prediction`
- guardrails de entrada e saida

Por isso, a avaliacao escolhida nao mede apenas um modelo de linguagem isolado,
mas o comportamento do sistema conversacional completo.

## Regras arquiteturais adotadas

As seguintes regras passam a orientar a avaliacao:

1. metricas tabulares e metricas generativas nao devem ser consolidadas em uma
   unica nota sem explicacao de contexto
2. a avaliacao da trilha LLM deve preferir o endpoint real `/llm/chat`
3. o golden set deve permanecer versionado e validado
4. mudancas em provider, prompt, tools, corpus ou guardrails devem ser tratadas
   como variaveis que podem alterar os resultados
5. artefatos de avaliacao devem ser persistidos em caminhos auditaveis

## Consequencias

### Positivas

- melhora a clareza da defesa tecnica do projeto
- evita confundir qualidade do modelo tabular com qualidade da trilha LLM
- aumenta auditabilidade por meio de artefatos e historicos
- aproxima a avaliacao generativa do comportamento real observado no serving

### Negativas

- aumenta o numero de artefatos e rotinas de avaliacao
- exige mais disciplina para interpretar resultados por camada
- a avaliacao generativa fica sensivel a mudancas no provider LLM
- a execucao end-to-end pode custar mais tempo do que testes locais isolados

## Alternativas consideradas

### Uma unica avaliacao consolidada

Rejeitada. Esse caminho esconderia diferencas entre performance supervisionada,
saude operacional e qualidade generativa.

### Avaliar a trilha LLM apenas por inspeção manual

Rejeitada. Isso reduziria reproducibilidade e dificultaria comparacao entre
execucoes.

### Avaliar apenas funcoes internas em vez do endpoint real

Rejeitada como estrategia principal. Embora testes unitarios continuem uteis, a
avaliacao principal deve passar pelo contrato real de `/llm/chat`.

## Implicacoes para evolucao futura

Evolucoes futuras devem considerar:

- expandir o golden set sem perder foco no dominio
- revisar metricas e criterios do judge quando o escopo do agente crescer
- reavaliar a estrategia caso o provider principal mude
- manter coerencia entre documentacao, tasks e artefatos produzidos
