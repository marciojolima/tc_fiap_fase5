# ADR-002 - Atendimento ao requisito de LLM servido via API com quantizacao aplicada

- Status: aceito
- Data: 02-05-2026

## Contexto

O projeto possui uma trilha conversacional com agente ReAct, RAG local e
provider LLM configuravel via `configs/pipeline_global_config.yaml`.

No estado atual do repositorio, o provider ativo e `claude`, com uso do modelo
`claude-haiku-4-5-20251001` via API da Anthropic:

```yaml
llm:
  active_provider: claude
  timeout_seconds: 60
  providers:
    claude:
      base_url: https://api.anthropic.com
      model_name: claude-haiku-4-5-20251001
      api_key_env_var: ANTHROPIC_API_KEY
      max_tokens: 1200
```

Esse caminho atende ao requisito de LLM servido via API, pois o projeto consome
um endpoint externo gerenciado. Porem, a quantizacao do modelo nao e
configuravel nem auditavel pelo projeto, ja que os pesos e o runtime pertencem
ao provedor externo.

O repositorio tambem preve um caminho alternativo com `Ollama`, incluindo
configuracao dedicada no arquivo global e `docker-compose.ollama.yml` para
subir o daemon e executar `ollama pull` do modelo configurado:

```yaml
llm:
  providers:
    ollama:
      base_url: http://127.0.0.1:11434
      model_name: qwen2.5:3b
```

No codigo, o provider `Claude` envia `model`, `messages`, `max_tokens` e
`temperature` para a API da Anthropic. Ja o provider `Ollama` envia `model`,
`messages` e `temperature` para `POST /api/chat`, usando `base_url`,
`model_name` e `timeout_seconds`.

## Decisao

Adotamos a seguinte interpretacao arquitetural para o requisito:

1. O caminho atual com `Claude` atende apenas a parte de "LLM servido via API".
2. O requisito completo de "LLM servido via API com quantizacao aplicada" so
   pode ser evidenciado de forma objetiva quando o provider ativo for local ou
   autogerenciado, como `Ollama`, com um modelo quantizado identificavel.
3. Para fins de documentacao e defesa tecnica, o projeto passa a registrar essa
   diferenca explicitamente nas ADRs.

## Justificativa

### Cenario atual com Claude

- O provider ativo hoje e `claude`.
- O projeto consome a API da Anthropic por meio de `base_url`,
  `model_name`, `api_key_env_var`, `max_tokens` e `timeout_seconds`.
- O runtime de inferencia e administrado pelo provedor externo.
- Nao ha parametro no projeto para selecionar nivel de quantizacao, formato de
  pesos ou variante quantizada do modelo Claude.

Conclusao: com `Claude`, o projeto nao consegue comprovar tecnicamente
"quantizacao aplicada pelo projeto", apenas o consumo de um LLM remoto servido
via API.

### Cenario alternativo com Ollama

- O projeto ja possui suporte nativo a `ollama` como provider LLM.
- O provider usa `base_url`, `model_name` e `timeout_seconds`.
- O compose dedicado sobe o servico `ollama`, espera healthcheck e executa
  `ollama pull` do modelo definido por `OLLAMA_MODEL`.
- Esse caminho permite apontar para variantes quantizadas de modelos abertos,
  como familias `Q4`, `Q5`, `Q8` ou equivalentes, desde que o nome do modelo
  configurado identifique essa variante.

Conclusao: com `Ollama`, o projeto pode atender ao requisito completo, desde
que utilize um modelo explicitamente quantizado e documente essa escolha.

## Parametros equivalentes validados no projeto

Embora `Claude` e `Ollama` usem superficies de configuracao parecidas, elas nao
sao identicas.

### Claude

```yaml
llm:
  active_provider: claude
  timeout_seconds: 60
  providers:
    claude:
      base_url: https://api.anthropic.com
      model_name: claude-haiku-4-5-20251001
      api_key_env_var: ANTHROPIC_API_KEY
      max_tokens: 1200
```

### Ollama

```yaml
llm:
  active_provider: ollama
  timeout_seconds: 60
  providers:
    ollama:
      base_url: http://127.0.0.1:11434
      model_name: qwen2.5:3b
```

Observacoes validadas no codigo:

- ambos usam `active_provider`, `base_url`, `model_name` e `timeout_seconds`
- apenas `Claude` exige `api_key_env_var`
- apenas `Claude` usa `max_tokens` na chamada implementada hoje
- `Ollama` depende adicionalmente do ambiente `OLLAMA_MODEL` no compose
  dedicado para executar o `pull` do modelo

## Consequencias

### Positivas

- a documentacao passa a distinguir claramente consumo de API externa e serving
  de modelo quantizado sob controle do projeto
- a defesa tecnica do requisito fica mais objetiva
- o projeto preserva flexibilidade entre provider remoto e provider local

### Negativas

- o cenario atual com `Claude` nao pode ser apresentado como evidencia plena de
  quantizacao aplicada
- para cumprir integralmente o requisito, seria necessario ativar `Ollama` com
  uma variante quantizada documentada

## Alternativas consideradas

### Manter apenas Claude como evidencia do requisito

Rejeitada. Esse caminho nao comprova quantizacao aplicada pelo projeto.

### Migrar a narrativa do requisito para Ollama

Aceita como alternativa documentada. O projeto ja possui base arquitetural para
isso, faltando apenas selecionar e registrar uma variante quantizada explicita
do modelo.

## Implicacoes para documentacao futura

As proximas ADRs ou documentos de arquitetura que tratem de provider LLM devem:

- deixar explicito qual provider esta ativo
- informar se a quantizacao e controlada pelo projeto ou pelo provedor externo
- registrar o `model_name` exato usado na operacao
- separar claramente "LLM via API" de "LLM quantizado sob gestao do projeto"
