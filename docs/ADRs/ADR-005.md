# ADR-005 - Atendimento ao requisito de RAGAS com 4 metricas calculadas e reportadas

- Status: aceito
- Data: 2026-05-02
- Relacionada a: [ADR-003](ADR-003.md), [ADR-004](ADR-004.md)

## Contexto

Um dos requisitos da trilha generativa do projeto e:

- RAGAS: 4 metricas calculadas e reportadas

Era necessario verificar se esse requisito foi de fato implementado no
repositorio, se estava apenas parcialmente coberto, ou se dependia apenas de
documentacao sem evidencia executavel.

## Verificacao realizada

A verificacao confirmou que o requisito esta implementado no projeto e possui
evidencia material no codigo, nos testes, nas tasks e nos artefatos gerados.

## Evidencias encontradas

### 1. Implementacao no codigo

O arquivo [src/evaluation/llm_agent/ragas_eval.py](../src/evaluation/llm_agent/ragas_eval.py)
implementa a avaliacao RAGAS end-to-end.

As quatro metricas configuradas no fluxo sao:

- `faithfulness`
- `answer_relevancy`
- `context_precision`
- `context_recall`

Essas metricas sao montadas explicitamente na lista `metrics` passada ao
`evaluate(...)` do RAGAS.

### 2. Avaliacao sobre o endpoint real

O fluxo nao avalia respostas artificiais ou funcoes isoladas. Ele monta o
dataset de avaliacao chamando o endpoint real `POST /llm/chat`, preservando o
comportamento completo da trilha:

- provider LLM ativo
- agente ReAct
- tools
- trace
- RAG
- guardrails

Isso reforca que a avaliacao esta implementada de forma operacional, e nao
apenas como stub de metricas.

### 3. Testes automatizados

O arquivo [tests/test_ragas_eval.py](../tests/test_ragas_eval.py) cobre partes
importantes do fluxo, como:

- carregamento do golden set
- chamada ao endpoint `/llm/chat`
- construcao do dataset de avaliacao
- adaptador de embeddings
- compatibilidade com o provider `Claude`

### 4. Tasks de execucao

O projeto expoe tasks dedicadas para executar o RAGAS:

- `poetry run task eval_ragas`
- `poetry run task eval_ragas_docker`
- `poetry run task eval_ragas_sample`
- `poetry run task eval_ragas_sample_docker`

Isso mostra que a avaliacao esta incorporada ao fluxo operacional previsto.

### 5. Artefatos reportados

O repositorio contem os artefatos esperados do RAGAS:

- `artifacts/evaluation/llm_agent/results/ragas_scores.json`
- `artifacts/evaluation/llm_agent/runs/ragas_runs.jsonl`

No arquivo `ragas_scores.json` ha um resultado persistido com as quatro metricas
presentes em `metrics_mean`.

## Decisao

Registramos que o requisito "RAGAS: 4 metricas calculadas e reportadas" esta
atendido no estado atual do projeto.

O atendimento e considerado completo do ponto de vista de implementacao, porque
ha:

- calculo das quatro metricas no codigo
- execucao prevista por task
- avaliacao sobre o serving real
- documentacao da estrategia
- artefatos persistidos com resultados

## Justificativa

### Por que consideramos o requisito atendido

O requisito nao pede apenas mencao conceitual ao RAGAS, mas metricas calculadas
e reportadas. O projeto satisfaz isso porque:

- calcula as quatro metricas explicitamente
- persiste a media dessas metricas em JSON
- mantem historico append-only das execucoes
- documenta como reproduzir o processo

### O que nao caracteriza implementacao parcial aqui

Nao se trata de implementacao parcial porque:

- as quatro metricas existem no codigo
- elas sao efetivamente agregadas e salvas
- o artefato final contem todas as quatro

Se existisse apenas o esqueleto da avaliacao, sem task, sem persistencia ou sem
as quatro metricas no resultado, o requisito seria parcial. Nao e o caso atual.

## Limitacoes e cuidados

Embora o requisito esteja atendido, a execucao depende de algumas condicoes
operacionais:

- serving ativo em `/llm/chat`
- provider LLM configurado e acessivel
- golden set valido
- ambiente com dependencias do RAGAS e embeddings disponiveis

Tambem e importante notar que a interpretacao das metricas continua sensivel a:

- mudancas no provider LLM
- mudancas no corpus do RAG
- mudancas no prompt e nas tools do agente
- mudancas no modelo de embeddings

Ou seja, o requisito esta atendido, mas os resultados nao devem ser tratados
como valores absolutos e imutaveis.

## Consequencias

### Positivas

- o projeto consegue defender tecnicamente a avaliacao da trilha RAG/LLM
- a avaliacao e reproduzivel por task
- os resultados ficam auditaveis por artefatos

### Negativas

- a execucao end-to-end depende de infraestrutura funcional
- a avaliacao pode ser mais lenta e custosa do que testes internos isolados

## Implicacoes para evolucao futura

Evolucoes futuras nessa trilha devem preservar:

- as quatro metricas obrigatorias
- a persistencia dos resultados
- o historico das execucoes
- a avaliacao sobre o endpoint real, sempre que possivel

Se houver mudanca estrutural no fluxo de avaliacao que remova alguma dessas
quatro metricas, esta ADR deve ser revisada.
