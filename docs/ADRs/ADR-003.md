# ADR-003 - Uso de agente ReAct com tools de dominio e RAG local

- Status: aceito
- Data: 02-05-2026
- Relacionada a: [ADR-002](ADR-002.md), [ADR-012](ADR-012.md)

## Contexto

O projeto expoe uma trilha conversacional em `POST /llm/chat`, com objetivo de
transformar predições, simulacoes e sinais operacionais em respostas mais
acessiveis para apoio a decisao no contexto de churn bancario.

Essa trilha nao substitui o modelo tabular. Ela atua como uma camada de
orquestracao sobre os ativos ja existentes do projeto:

- endpoint tabular `/predict/raw`
- simulacao de cenarios de negocio
- artefatos de monitoramento de drift
- documentacao e metadados recuperados por RAG local

No estado atual do repositorio, a decisao primaria de provider LLM e o uso de
`Claude` via API da Anthropic, conforme registrado em
[ADR-002](ADR-002.md). Essa escolha impacta diretamente o desenho do agente,
porque o loop de execucao depende de respostas estruturadas em JSON para
decidir entre chamar uma tool ou concluir a resposta final.

O projeto tambem possui uma alternativa com `Ollama`, mas a arquitetura do
agente e independente do provider. O contrato esperado do provider e simples:

- receber uma lista de mensagens
- retornar texto
- opcionalmente expor metadados de modelo e provider

## Decisao

Adotamos um agente do tipo ReAct, implementado localmente no projeto, com as
seguintes caracteristicas:

1. o endpoint `POST /llm/chat` executa um loop compacto de pensar, agir e
   observar
2. a orquestracao das tools fica no proprio codigo do repositorio, sem
   dependencia de framework externo de agentes
3. o conjunto padrao de tools do agente e:
   - `rag_search`
   - `predict_churn`
   - `drift_status`
   - `scenario_prediction`
4. perguntas documentais devem priorizar `rag_search` antes da resposta final
5. perguntas operacionais ou de negocio devem usar tools especificas do dominio
   quando necessario, em vez de inferencia livre do modelo
6. o provider LLM primario para esse agente e `Claude`, conforme a ADR-002

## Justificativa

### Por que ReAct

O padrao ReAct atende bem ao problema do projeto porque separa raciocinio,
execucao de ferramenta e resposta final. Isso permite:

- reduzir alucinacao em perguntas sobre rotas, arquivos, configuracoes e
  artefatos do repositorio
- usar o modelo tabular real quando a pergunta exige predicao de churn
- usar a simulacao de cenarios quando a pergunta envolve comparacao de impacto
- consultar o estado real de drift sem depender de memoria do modelo

No codigo, essa decisao aparece no loop de `run_react_agent`, que exige saida
JSON valida, rastreia `trace`, registra `used_tools` e limita o numero de
iteracoes.

### Por que um agente local e nao um framework externo

O projeto implementa o agente diretamente em `src/agent/react_agent.py`.
Escolhemos esse caminho porque ele:

- preserva baixo acoplamento
- facilita auditoria da logica de decisao
- deixa explicita a politica de uso das tools
- simplifica testes unitarios e integracao com o endpoint `/llm/chat`

Como o problema do projeto tem escopo controlado, o custo de manter um executor
ReAct proprio e aceitavel e traz mais transparencia do que uma abstracao mais
pesada.

### Por que essas tools

O conjunto atual de tools cobre as necessidades centrais da trilha
conversacional.

#### `rag_search`

Recupera contexto do repositorio e da documentacao local via RAG. O desenho do
indice vetorial em memoria com cache local, sem vector store externo, esta
formalizado em [ADR-012](ADR-012.md). `rag_search` e a tool prioritaria para
perguntas documentais sobre:

- rotas da API
- arquivos e configuracoes
- arquitetura e operacao do projeto
- documentacao tecnica e artefatos

#### `predict_churn`

Executa predicao de churn a partir de um payload JSON equivalente ao contrato de
`/predict/raw`. Isso evita respostas especulativas quando o usuario fornece um
cliente ou um payload de inferencia.

#### `drift_status`

Le o status mais recente de monitoramento em
`artifacts/evaluation/model/drift/drift_status.json`. Isso permite ao agente
responder sobre saude operacional e drift usando evidencia real do projeto.

#### `scenario_prediction`

Executa inferencia de cenarios, inclusive comparacoes entre `baseline_scenario`
e `improved_scenario`. Isso torna o agente util para discussoes de retencao,
impacto de ajustes em atributos do cliente e simulacoes orientadas a negocio.

## Regras arquiteturais adotadas

As seguintes regras fazem parte da decisao:

1. o agente deve sempre responder em JSON internamente durante o loop ReAct
2. a resposta final ao usuario deve ser em portugues do Brasil
3. o agente nao deve inventar fatos sobre routes, files, configs, tools ou
   artefatos sem contexto observado
4. perguntas classificadas como documentais devem restringir o conjunto de
   tools para `rag_search`, quando essa tool estiver disponivel
5. entrada e saida passam por guardrails antes de chegar ao usuario final
6. a API deve devolver `trace` e `used_tools` quando solicitado, para facilitar
   depuracao e avaliacao

## Relacao com ADR-002 e ADR-012

A [ADR-002](ADR-002.md) registrou que a decisao primaria de provider LLM e o
uso de `Claude` via API, embora o projeto mantenha alternativa com `Ollama`.

Esta ADR depende dessa escolha por tres motivos:

1. o agente precisa de boa aderencia a instrucoes para produzir JSON valido em
   todas as iteracoes
2. o agente precisa alternar de forma confiavel entre chamada de tool e
   resposta final
3. a qualidade da experiencia conversacional depende da estabilidade do
   provider configurado em `/llm/chat`

Ja a [ADR-012](ADR-012.md) registra a decisao de infraestrutura da camada de
RAG, explicando por que o projeto usa indice vetorial em memoria com cache
local, sem vector store externo.

Por isso, essas ADRs devem ser lidas em conjunto:

- a ADR-002 decide como o projeto consome o modelo
- a ADR-003 decide como esse modelo e usado para orquestrar tools e RAG
- a ADR-012 decide como o RAG e materializado no runtime

Se no futuro o provider principal mudar de `Claude` para `Ollama`, a arquitetura
do agente pode ser mantida, mas a validacao de robustez para saida JSON e
latencia operacional deve ser reavaliada.

## Consequencias

### Positivas

- respostas ficam mais ancoradas em evidencias reais do repositorio e dos
  artefatos do projeto
- o agente reaproveita ativos existentes em vez de duplicar logica de negocio
- `trace` e `used_tools` aumentam auditabilidade
- a arquitetura suporta tanto provider remoto quanto provider local

### Negativas

- o agente depende da capacidade do provider LLM de seguir o contrato de JSON
- ha aumento de complexidade em relacao a um chat direto sem tools
- perguntas fora do escopo das tools podem exigir mais iteracoes ou fallback
  para resposta final sem acao
- a heuristica de pergunta documental pode restringir demais alguns casos
  ambiguos

## Alternativas consideradas

### Resposta direta sem agente

Rejeitada. Esse caminho reduziria a capacidade de usar predicao real, simulacao,
RAG e monitoramento de forma controlada.

### Agente com apenas RAG

Rejeitada. O projeto precisa ir alem de recuperacao documental e acionar
predicao, cenarios e monitoramento.

### Framework externo de agentes

Rejeitada no momento. Para o escopo atual, a implementacao propria oferece mais
controle, menor acoplamento e melhor auditabilidade.

## Implicacoes para evolucao futura

Evolucoes futuras nessa trilha devem considerar:

- adicionar novas tools apenas quando houver fonte de evidencia concreta no
  projeto
- manter a separacao entre perguntas documentais, de predicao, de cenario e de
  monitoramento
- reavaliar a decisao se o provider principal deixar de atender bem ao contrato
  de JSON estruturado
- atualizar esta ADR caso o conjunto padrao de tools mude
